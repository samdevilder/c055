    1. Idempotent in this case means that the method no matter how many times it is called by the client, the result from the server will not change. This only applies to the result or resource's representation, as a idempotent could still possibly change the resource.
    2. Idempotent requests simplify the implementation of a caching HTTP proxy because idempotent methods are always stored on any middleware proxies that are between the client and the server.
    3. Visitor counters often break this assumption because every time that a safe method like the GET method is called for the visitor counter, the client side isn't concerned with the side effects that it will have on the resource. The server however has to decide if the GET method should be an indicator to increment the visitor counter while responding to the GET request or if it should simply respond to the request without having to change anything about the resource.
    4. Such counters could be negatively affected by the implementation of proxies because idempotent methods could possibly change the resource of the counter, but because it's an idempotent method, the change in the resource will only be saved in the middleware proxies between the user and server. Now the resource on the cache is different than that on the server. This would prove inconvenient for the counter on the main server as it will now have changes to the resource on all these different proxy caches due to the behavior of idempotent methods with resource changes.
  2. A stateless protocol is a communications protocol that treats each request as an independent transaction that is unrelated to any previous request so that the communication consists of independent pairs of request and response.
    Pro
    Simplifies server design because there is no need to dynamically allocate storage to deal with conversations in progress. If a client session dies in mid-transaction, no part of the system needs to be responsible for cleaning up the present state of the server.
    Con
    It may be necessary to include additional information in every request and this extra information will need to be interpreted by the server.

The 303 response MUST NOT be cached, but the response to the second (redirected) request might be cacheable.
